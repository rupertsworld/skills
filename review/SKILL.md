# Review

A GTD-inspired review process. One workflow, scaled by depth. Ask the user whether this is a daily or weekly review if unclear.

Refer to any provided documentation for specifics on where tasks, calendars, plans, project notes, someday lists, and reminders live.

## The process

### 1. Get Clear — clean up before looking forward

Clear the decks so the review starts from a clean state:
- **Yesterday:** Quick review — anything unfinished that needs to carry forward?
- **Checked todos:** Sweep the task list for completed items (checked checkboxes) and remove them.
- **Weekly:** Go deeper — clear out stale waiting-on items, resolve any ambiguous tasks.

Inbox processing is handled separately — don't fold it in here.

### 2. Collect — get current state into view

Gather everything in parallel:
- **Task list** — current next actions, waiting-on items, active projects
- **Planning, project notes, and someday lists** — read all of them, even for daily reviews. These are the source of truth for what's important beyond this week.
- **Calendar events** — check all available calendars, not just the primary. Today + tomorrow for daily, 7 days ahead for weekly.
- **Reminders** — anything due or upcoming

### 3. Process — clarify and clean up

Walk through what's collected:
- **Tasks:** Is each action still relevant? Is it actually a single next action or a project in disguise? Anything done that should be removed? Anything in Waiting that's resolved?
- **Calendar:** Any meetings that need prep? Conflicts? Deadlines?
- **Overdue sweep:** Check for past/overdue dated tasks. For each one, ask the user: bump to today, convert to an undated next action, or drop?
- **Reminders:** Anything completed or stale?
- **Planning sweep:** For each area/project file, are there items that are now timely? Promote to the task list or schedule on the calendar. (Weekly goes deeper here, but daily should still check.)
- **Weekly — Projects check-in:** For each active project, what's the status? Is there a clear next action? Is anything stuck or stalled?

Present findings as a concise summary — bullet points, not paragraphs.

### 4. Organize — put things where they belong

With the user, decide what to do with each item surfaced:
- **Do:** stays as a next action on the task list
- **Schedule:** goes on the calendar with a date/time block
- **Delegate/Waiting:** moves to waiting with who/what you're waiting on
- **Defer:** goes back to a planning or someday list for later
- **Drop:** remove it

### 5. Prioritize — decide what matters now

This is a conversation, not a dump. Surface candidate items from priorities, projects, and the action list, then discuss with the user what should make today's/this week's cut.

- **Daily:** Propose 3-5 items based on deadlines, momentum, and what's highlighted. Ask the user to confirm, swap, or add.
- **Weekly:** 3-5 key outcomes for the week, informed by long-term goals and planning files.

Consider: What has a hard deadline? What would be most impactful? What's blocking someone else? What has momentum from yesterday? What connects to longer-term goals?

### 6. Update — make the changes

With the user's agreement, update:
- Task list — write priorities, reorder actions, clean up completed items
- Calendar — schedule time blocks, dated tasks, and reminders
- Planning and project files — if status changed or priorities shifted
- Reminders — if something needs a push notification

Always confirm before writing changes.

### 7. Update the task list

**Daily → update the Today section:**

Update the Today section at the top of the task list. This replaces any previous Today content. Refer to any provided documentation for formatting conventions.

The Today section should read like a trusted aide giving a quick debrief — opinionated, short, focused on what matters. Items placed in Today are moved out of the main actions list (no duplication).

**Weekly → rebuild the entire To-Do file:**

Reconstruct the full task list from scratch based on the review. Rewrite Priorities, Actions, Waiting, and Projects sections. Emphasis on aligning with long-term goals. Also populate the calendar with time blocks and dated tasks for the week ahead.

## Cadences

- **Daily:** Read planning, project, and someday files upfront for context. Clean up, collect, then focus on constructing Today from the existing task list. Light touch on steps 3-4, emphasis on step 5 (prioritize) and step 7 (Today section). Also update calendar as needed.
- **Weekly:** The full process. Rebuild the entire task list. Go deep on planning, projects, and long-term goals. Populate the calendar with time blocks and tasks for the week. More emphasis on what's important over what's urgent.

## Critical engagement

The review is not just administrative — it should challenge and hold the user accountable. Refer to any provided user context documentation for longer-term values and goals.

During steps 3-5, actively assess:

- **Am I on track?** Check stated priorities and deadlines against actual progress. If something was supposed to ship this week and hasn't started, say so directly.
- **Are projects on track?** For each active project, is there forward motion or is it stalled? What's actually blocking it vs. what's just not getting attention?
- **What needs to happen ASAP?** Distinguish between genuinely urgent (deadlines, blocking others, time-sensitive) and merely lingering.
- **What can be dropped or deferred?** If the action list is growing faster than it's shrinking, suggest items to cut. Be honest about what's been sitting there for weeks untouched — it might not be a priority anymore.
- **Does the week/day align with what matters?** Cross-reference against longer-term goals. If the user is spending all their time on admin and none on the work that matters to them, flag it.

Be direct. "This has been on the list for three weeks — is it still a priority or should it move to a someday list?" is better than silently carrying it forward.

## Guidelines

- Be concise. Surface the most important items first.
- Always confirm before making changes to files or calendar.
- Read planning, project, and someday files at the start of every review — they provide essential context for prioritization.
- **Update project and planning files too** — when status changes or items are completed, update the relevant files, not just the task list. These are the durable record.
- **Projects must have next actions.** During step 3 (Process), scan the Projects section and verify every project has a corresponding next action on the Actions list (or is explicitly in Waiting). If a project has no clear next action, flag it and ask the user.
- **Save context at end of review.** After making updates, persist any relevant information surfaced during the conversation to the current context tracking doc (refer to any provided documentation for its location). Update with new context, remove stale entries. This ensures continuity between sessions.
